# 增量合并策略 — Incremental Merge Strategy

## 核心原则

**永远不覆盖，只追加（Never overwrite, always append）。**

所有来自记忆系统（working/longterm/emotional）和进化系统（correction/drift）的数据，在合并到 soul.md 时必须遵循追加原则。已有内容保持不变，新增内容以结构化注释标记来源和时间。

**理由：**
- 保留认知演变的历史轨迹
- 便于回溯"这个结论是什么时候得出的"
- 避免因新数据覆盖旧数据而丢失重要背景
- 支持多层记忆并存（冷却中的旧数据 + 激活的新数据）

---

## 追加元数据格式

所有追加内容必须包含以下标记：

```html
<!-- [追加于 {YYYY-MM-DD}，来源：{source}，置信度：{confidence}] -->
```

**来源标识对照表：**

| 来源标识 | 含义 |
|----------|------|
| `correction#{n}` | 来自某条纠正记录 |
| `session#{id}` | 来自某次会话的工作记忆晋升 |
| `memory-confirm#{n}` | 来自长期记忆的多次确认 |
| `emotional-trigger` | 来自情感记忆的触发事件 |
| `drift-review` | 来自漂移检测的自查结果 |
| `user-explicit` | 用户明确告知 |
| `inference` | 基于上下文的合理推断（需标注 confidence） |

## 按数据类型的合并规则

### 类型 1：新体验 / 新事实

**来源：** working memory 晋升
**合并方向：** → `memory.md` 时间线 + 评估是否合并到 soul.md

```markdown
<!-- 新增内容（在现有时间线末尾追加）-->
## {date} — 新增记忆

<!-- [追加于 {date}，来源：session#{id}，置信度：{conf}] -->
{新记忆内容}
```

**评估标准：** 同一话题在 3+ 会话中确认 → 触发向 soul.md 的合并建议

### 类型 2：新证据支持现有结论

**来源：** 多条独立来源指向同一结论
**合并方向：** → 追加到 soul.md 现有章节，新证据在前，旧证据在后保留

```markdown
{现有结论}
<!-- [补充证据：{date}，来源：{source}，置信度：{conf}] -->
<!-- [原有证据：{old_date}，来源：{old_source}，置信度：{old_conf}] -->
```

**规则：** 保留旧证据不删除，置信度取所有证据的平均值，新证据作为最新参照。

### 类型 3：矛盾证据

**来源：** 用户纠正或新信息推翻旧记忆
**合并方向：** → 写入 `conflicts.md`，不覆盖 soul.md 中的旧内容

```markdown
## 冲突记录 #{n} — {date}

**类型**: 矛盾
**条目 A（现有内容）**: {soul.md 中的原文}
**条目 B（新证据）**: {用户纠正/新信息}
**冲突点**: {具体矛盾所在}
**解决状态**: {pending | resolved}
**解决方式**: {user_correction | observation_override | 需要用户确认}
```

**规则：** 矛盾不直接删除旧内容。新内容在冲突解决前以 `[?]` 前缀标记，待确认后移除前缀。

### 类型 4：新价值观 / 新信念

**来源：** 用户表达的人生观、原则、优先级变化
**合并方向：** → 追加到 `soul.md` 的 cognition 章节

```markdown
<!-- 在 cognition 章节末尾追加 -->
### {date} — 新增信念

<!-- [追加于 {date}，来源：{source}，置信度：{conf}] -->
{信念内容描述}

**表达场景**：{用户是在什么情况下表达这个信念的}
```

**注意：** 价值观变化是渐进式的，旧信念保留，新信念新增。两者可能并存（如"以前我觉得 X，现在我认为 Y"），这是正常的。

### 类型 5：新语言习惯 / 口头禅

**来源：** correction（Layer 2）或会话观察
**合并方向：** → 追加到 `soul.md` 的 expression 章节

```markdown
## 语言习惯补充

<!-- [追加于 {date}，来源：correction#{n} / session#{id}] -->
- {新发现的语言习惯}（观察到于 {date}）
```

**合并条件：** 同一语言习惯在 2+ 不同会话中出现，或被用户明确确认。

### 类型 6：新行为模式

**来源：** 多次观察或纠正（Layer 4）
**合并方向：** → 追加到 `soul.md` 的 behavior 章节

```markdown
## 行为模式补充

<!-- [追加于 {date}，来源：{source}] -->
- {新发现的行为模式}（观察到于 {date}）
**观察频率**: {n} 次
```

### 类型 7：新情感触发器

**来源：** emotional.md 的情感记忆晋升
**合并方向：** → 追加到 `soul.md` 的 emotion 章节

```markdown
## 情感触发器补充

<!-- [追加于 {date}，来源：emotional-trigger] -->
- 触发话题：{topic}
  情感反应：{reaction}
  应对偏好：{coping_preference}
  隐私级别：{sensitive | core-wound}
```

**注意：** 隐私级别为 `core-wound` 的触发器，在合并时需额外标注 `{不主动唤醒}`。

### 类型 8：新边界

**来源：** 纠正（Layer 4）或边界突破事件
**合并方向：** → **前置**到 `soul.md` Layer 0（边界章节）

```markdown
## 边界规则补充

<!-- [追加于 {date}，来源：correction#{n} / boundary-violation] -->
- **{date} 新增**：{边界描述}
  触发场景：{什么情况下会触发这个边界}
  违反后果：{用户表达的负面后果}
```

**重要：** 新边界必须**前置**（放在 Layer 0 章节开头），而不是追加到末尾，确保边界规则按时间倒序排列，最近的边界最显眼。

## 三级合并优先级

当多个来源同时要求合并时，按以下优先级处理：

```
优先级 1 — 用户显式纠正（correction confirmed）
  → 立即应用，优先级最高

优先级 2 — 情感记忆触发（emotional.md）
  → 立即应用，涉及隐私保护的内容需特殊处理

优先级 3 — 长期积累（memory.md 多次确认）
  → 批量处理，每周或每 10 条新确认后合并一次
```

**批量合并的时机：**
- 同一 soul.md 章节积累 5+ 条待合并内容
- 每次快照创建前
- 用户主动询问"你学到了什么关于我的事"

## 合并前的检查步骤

每次合并前，执行以下检查：

```markdown
## 合并前检查清单

[ ] 1. 确认数据来源可靠（至少 confidence ≥ 0.7，或有用户显式确认）
[ ] 2. 确认目标 soul.md 章节存在
[ ] 3. 检查是否有现有内容与新内容矛盾（→ 写入 conflicts.md）
[ ] 4. 确认追加格式正确（包含元数据注释）
[ ] 5. 确认新内容放在正确位置（前置/末尾/章节内）
[ ] 6. 如果涉及 core-wound，标注 {不主动唤醒}
[ ] 7. 合并后更新 soul.md 的"最后更新"字段
```

---

## 合并冲突的处理

当需要合并的内容与 soul.md 中现有内容**直接矛盾**时：

```
Step 1: 不直接覆盖
Step 2: 将矛盾写入 conflicts.md
Step 3: 在 soul.md 中给矛盾内容加注：
        「[待确认，{date} 起与此来源矛盾：{source}]」
Step 4: 等待用户纠正或进一步确认
Step 5: 用户确认后，按确认结果处理（移除待确认标注或更新内容）
```

## 合并记录

每次合并后，在 `data/merge-log.md` 中记录：

```markdown
## Merge #{n} — {timestamp}

**合并来源**: {source_files}
**合并目标**: {soul.md sections}
**合并条目数**: {n}
**覆盖章节**: {sections}

### 合并详情
- {条目摘要 1} → {目标章节}
- {条目摘要 2} → {目标章节}

### 冲突处理
- {冲突描述} → 写入 conflicts.md #{conflict_id}

**合并后状态**: soul.md 各章节已更新
**下次合并窗口**: {预计下次合并时间或触发条件}
```

## 何时更新 soul.md vs 何时保持分离

```
更新 soul.md 的情况：
✓ 同一结论在 3+ 会话中得到独立确认
✓ 用户显式纠正了 soul.md 中的内容
✓ 情感记忆中的模式已稳定（不再是临时情绪反应）
✓ 漂移检测发现 soul.md 基准已明显偏离实际

保持分离（留在 memory.md / emotional.md）的情况：
✗ 新发现但只确认 1-2 次
✗ 用户表示"这个可能不是固定的，以后可能变"
✗ 涉及高度隐私的内容（留在专用文件中更安全）
✗ 实验性观察（待更多数据验证）
```
