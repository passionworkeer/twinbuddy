## 前言：关于 Demo 阶段的说明

**重要提醒：** 本产品目前为 **黑客松 Demo 阶段**，使用的数据为模拟数据，而非抖音真实数据。

所有 Demo 环节展示的效果，在未来接入抖音真实数据后，会有**质的飞跃**。

## 一、产品定位类

### Q1：你们这个产品是做什么的？

**A：**

TwinBuddy 是一个基于数字孪生体的智能搭子匹配产品。

我们的核心体验不是"给你一个工具，你自己去找搭子"，而是——

> **你正常刷抖音，突然一张 AI 卡片弹出来：你的数字孪生体已经帮你和对方聊过了，行程排好了，搭子也筛出来了，你只需要点"同意"。**

两个数字孪生体在后台替你谈判，你只需要旁观、点头。这就是"刷到懂你的瞬间"。

---

### Q2：和现有的搭子产品有什么区别？

**A：**

| 传统搭子产品 | TwinBuddy |
|-------------|-----------|
| 你主动找、自己聊 | 数字孪生体在后台替你谈 |
| 靠问卷了解你 | 数字孪生体模仿你的真实人格 |
| 你承担沟通成本 | 你只需要旁观、点头 |
| 事后匹配 | 刷到即触发 |

本质上，传统产品是"工具"，我们是"服务"——你不用动手，结果端到你面前。

---

### Q3：你们这个想法是怎么来的？

**A：**

我们观察到两个趋势：

**趋势一：每个人都在抖音上留下大量真实人格数据**
- 你点赞什么、评论什么、收藏什么——这些比任何问卷都真实
- 问题是这些数据没有被用起来

**趋势二：AI Agent 正在接管"跑腿"工作**
- 从"我自己做"到"AI 帮我做"
- 但大多数 AI 还停留在"回答问题"，而不是"替你行动"

我们的想法是：**把抖音的真实人格数据和 AI Agent 结合起来，让你的数字孪生体替你处理一切"需要找个伴"的事。**

## 二、技术类

### Q4：数字孪生体是怎么构建的？

**A：**

我们使用 **MING 人格蒸馏引擎**，把用户数据转化为数字孪生体。

核心是四维蒸馏：

| 维度 | 说明 |
|------|------|
| cognition（认知）| 思考方式、决策风格 |
| expression（表达）| 说话语气、措辞习惯 |
| behavior（行为）| 行为模式、行动偏好 |
| emotion（情感）| 情感反应、情绪边界 |

数据来源不同，数字人的"像你程度"也不同——这是下一个问题的重点。

---

### Q5：两个数字人是怎么协商的？

**A：**

我们使用 **LangGraph 状态机** 控制协商流程。

```
IDLE → PERSONA_INIT → CHAT_ROUND → CONFLICT_DETECTED →
NEGOTIATION → CONSENSUS_FOUND → REPORT_GENERATED → CARD_TRIGGERED
```

具体来说：

1. 你的数字人和对方的数字人，各自作为一个 Agent
2. 双方轮流发言，提出提案、评估对方、做出让步
3. 如果检测到冲突（如"你爱临时改，对方要提前锁定"），进入专门协商阶段
4. 最终输出：兼容度分数 + 分维度报告 + red_flags 预警 + 协商结论

评委可以现场看到——两个 AI 真的在"讨论"，而不是简单匹配。

---

### Q6：你们的 Agent 技术有什么亮点？

**A：**

三个核心技术点：

**① 固定权重融合**
```
MBTI（性格） 权重 0.9 — 最高，因为它最稳定
说话风格示例 权重 0.8 — 直接决定语气
兴趣标签    权重 0.7 — 结构化偏好
```
数据不足时标记 confidence: "low"，避免过度自信。

**② 过程可见**
协商过程不是黑盒——用户可以看到两个数字人怎么聊、怎么让步、怎么达成共识。这本身就是一种"懂你"的体验。

**③ 预生成兜底**
如果 LLM 调用超时或失败，预生成的 Mock 结果可以立即降级展示，保证 Demo 流畅。

## 三、数据与隐私类

### Q7：你们用了什么数据？用户隐私怎么保护？

**A（诚实回答）：**

> **我们现在是 Demo 阶段，用的是模拟数据。**

具体来说：
- MBTI 是用户手动选的
- 兴趣标签是用户手动点的
- 说话风格是用户自己输入的
- 对方人格来自 20 个预设 Mock

**没有用任何真实的抖音用户数据。**

未来如果接入抖音数据，我们的设计是：
- **数据隔离**：用户人格数据存在本地（localStorage），不上传到服务器
- **最小化原则**：只收集人格蒸馏必需的数据
- **用户控制**：用户可以随时清空自己的数字孪生体

隐私是数字孪生体产品的核心问题，我们的设计从一开始就考虑了这一点。

---

### Q8：未来接入抖音数据后，效果会有什么提升？

**A（重点回答）：**

> **这可能是今天最重要的一个问题。**

**当前 Demo vs. 抖音真实数据：**

| 数据维度 | Demo 阶段 | 接入抖音数据后 |
|---------|-----------|--------------|
| 性格推断 | MBTI 16型（粗粒度）| **多维行为数据（细粒度）** |
| 兴趣偏好 | 3-6个手动标签 | **点赞/收藏/评论/观看时长多维分析** |
| 说话风格 | 用户自述（可能失真）| **评论/弹幕/私信真实语料** |
| 决策风格 | MBTI 推断 | **浏览行为/购买决策真实反映** |
| 人际偏好 | 标签推断 | **社交互动数据真实建模** |

**抖音本身就是一个巨大的用户人格数据矿：**

```
你的每一次互动，都在告诉抖音"我是谁"：
├── 你看了什么视频 — 内容偏好
├── 你点了什么赞 — 情绪共鸣点
├── 你评论了什么 — 你怎么表达
├── 你收藏了什么 — 你认为有价值的东西
└── 你搜索了什么 — 你主动寻找的

这些数据 → 训练 → 更像你的数字人
```

**举一个具体例子：**

Demo 阶段，你说"我喜欢旅行"——我们只能推断你是个爱旅行的人。

接入抖音数据后，我们看到：
- 你连续看了 20 个川西视频
- 你点赞了"说走就走"类型的 vlog
- 你评论过"我也想去，但怕高反"
- 你收藏了一个"川西装备清单"

这些组合起来，你的数字人就不是"爱旅行的人"，而是——**一个向往川西、担心高反、偏好说走就走风格、会在出发前做攻略的 ENFP 型用户**。

这就是质的飞跃。

---

### Q9：数字孪生体是永久的吗？数据归谁？

**A：**

**当前 Demo 设计：**
- 数据存在浏览器 localStorage
- 关闭浏览器即清空
- 每次访问都是新的开始

**未来产品设计（愿景）：**
- 用户拥有自己数字孪生体的完整控制权
- 可以导出、删除、重新训练
- 数据所有权归用户，不归平台

## 四、商业与落地类

### Q10：这个产品能落地吗？商业价值在哪里？

**A：**

我们有清晰的落地路径：

**短期（本次黑客松 → 大区赛 → 总决赛）：**
- 验证核心体验：数字孪生体 + 协商匹配
- 打磨产品形态：独立 App / 小程序

**中期（接入抖音数据后）：**
- 抖音本身就是一个分发渠道
- 数字孪生体可以赋能抖音的社交功能
- 用户不需要下载新 App，在抖音内即可体验

**商业价值：**
- **提升用户粘性**：你的数字孪生体越像你，你就越离不开抖音
- **开拓社交场景**：抖音做搭子匹配，有数据优势
- **数据价值闭环**：用户在平台留下更多数据 → 数字人更准 → 用户更依赖

---

### Q11：为什么要做"旅游搭子"这个场景？

**A：**

两个原因：

**第一：旅游搭子是强需求、高摩擦**

强需求——每个人都找过/被找过搭子。
高摩擦——需要反复沟通：去哪、什么时候、怎么去、住哪、吃什么……

数字孪生体正好解决高摩擦问题。

**第二：旅游搭子是最佳展示场景**

旅游决策是多维度的（时间/地点/预算/节奏/美食偏好……），可以充分展示：
- 数字孪生体的"懂你程度"
- 协商过程的复杂度
- 兼容度报告的价值

但这个框架不只旅游——美食、健身、演唱会、学习搭子，都适用。

---

### Q12：这个产品和抖音是什么关系？

**A：**

我们是**赛道三 AI体验**的参赛作品，核心展示的是：

> **在抖音的信息流中，创造一种新的内容单元——它不是被观看的视频，而是"刚好懂你"的瞬间。**

**长期愿景：**

如果有机会接入抖音真实数据，数字孪生体可以：
1. 基于抖音的观看/点赞/评论数据训练，更像用户
2. 出现在抖音 Feed 的信息流中，作为一种新的内容形态
3. 赋能抖音现有的社交功能（群组、活动、直播连麦……）

## 五、产品体验类

### Q13：用户为什么需要旁观两个 AI 聊天？直接给结果不行吗？

**A：**

好问题。这也是我们设计时反复思考的问题。

**直接给结果的缺点：**
- 黑盒感：用户不知道为什么给你推荐这个搭子
- 不信任：没有过程，用户很难相信结果
- 无惊喜：体验是功能性的，没有情感共鸣

**旁观对话的价值：**
- **可解释性**：你看到"我的数字人说她7点起，所以我让了步"——协商过程就是推荐理由
- **信任感**：过程可见，用户更愿意相信结果
- **情感共鸣**："另一个我"真的在帮我说话，这种体验很独特
- **参与感**：你不是在用工具，你是在"看戏"——看自己的数字人替自己谈判

就像看两个自己的分身在讨论，你不需要介入，只需要点头或摇头。

---

### Q14：兼容度分数是怎么算出来的？

**A：**

四维对比：

| 维度 | 对比内容 |
|------|---------|
| 旅行节奏 | 随性 vs. 计划型 |
| 消费观 | 体验优先 vs. 性价比为王 |
| 沟通风格 | 直接 vs. 喜欢事前讨论 |
| 独处/社交 | 需要独处 vs. 喜欢热闹 |

每个维度 0-100 分，加权平均得到总分。

冲突的地方单独标注，并给出"磨合建议"——比如"大框架定，细节留白"。

---

### Q15：如果协商失败怎么办？

**A：**

三种情况：

**情况一：3轮内达成共识**
→ 兼容度报告显示结论，用户决定是否同意

**情况二：3轮后仍有冲突**
→ 显示"仍在协商中"，给出分维度报告
→ 标注关键冲突点
→ 用户可以选择"继续协商"或"放弃"

**情况三：LLM 超时/失败**
→ 预生成的 Mock 结果立即降级展示
→ 保证 Demo 流畅，不翻车

## 六、挑战与局限类

### Q16：你们现在最大的挑战是什么？

**A：**

**当前 Demo 阶段：**
- 数据粒度太粗（MBTI 只有16型，说话风格靠自述）
- 协商轮次有限（最多3轮）
- Mock 人格池不够丰富（只有20个）

**核心挑战是数据，不是技术。**

技术路线是清晰的：Fusion Engine + LangGraph 状态机 + 四维蒸馏。
数据才是瓶颈——数字人像不像你，取决于输入数据丰不丰富。

---

### Q17：这个产品的局限性在哪里？

**A：**

坦诚地说：

**① 数字人的"像你程度"依赖数据质量**
如果用户不活跃，或者数据噪声大，数字人可能失真。

**② 协商结果的可靠性**
LLM 生成的协商结论不一定总是合理的，需要有人工校验机制。

**③ 隐私与信任**
用户是否愿意让平台拥有自己的数字孪生体？这需要清晰的隐私政策和用户教育。

**④ Demo vs. 真实产品**
现在展示的一切，在真实场景中需要考虑：性能、并发、安全、监管……

我们正视这些局限，每一个都是未来需要解决的问题。但 Demo 的价值在于验证核心假设——如果体验方向是对的，局限是可以被克服的。

---

### Q18：评委质疑"这不是真正的抖音，怎么体现差异化"？

**A：**

我们坦诚：这是 Demo，不是抖音内嵌功能。

但 Demo 验证的是**产品方向**：

> **如果这个体验是用户喜欢的，那么把它放进抖音的真实环境会更好——有真实数据、有分发渠道、有用户基础。**

我们今天展示的是"可能性"：
- 数字孪生体 + 协商匹配这个体验是有效的
- 用户愿意"旁观"而不是"自己动手"
- 兼容度报告 + 协商过程增加了信任感和参与感

这些验证完成后，接入抖音数据就是顺水推舟。
