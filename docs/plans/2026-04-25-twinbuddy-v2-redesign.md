> 日期：2026-04-25
> 状态：初稿，待评审

## 一、产品愿景与核心定位

**一句话定位：** AI 驱动的旅行搭子双边匹配平台。

核心差异：把最难的一步——「判断对方是否真的合适」——前置给 AI 数字分身完成。用户不需要刷 100 个人工筛选，数字人代理已经帮你谈好了。

**产品哲学：「数据是一切」**
每个功能触点都在采集用户数据：
- 发帖 / 评论 → 偏好暴露
- AI 助手对话 → 语言风格 + 旅行偏好
- 数字人主动探查 → 结构化偏好补充
- 协商过程 → 让步模式 + 冲突容忍度
- 6 轮盲选游戏 → 决策风格 + 价值观差异

所有数据最终服务于让数字分身越来越精准，协商质量持续高于任何后来者。

## 三、Onboarding 流程

### 流程概览
```
启动 → 欢迎页 → 快速问卷（~3分钟） → AI助手开场访谈（~5分钟） → 主页
```

### 快速问卷内容（3-5题，极简）

| # | 问题 | 目的 |
|---|------|------|
| 1 | 你的 MBTI 是？（16选1） | 人格锚点，影响协商风格 |
| 2 | 你通常去哪里旅行？（多选：同城/周边城市/国内/国外） | 目的地范围 |
| 3 | 你的旅行预算区间？（单选：穷游/经济/舒适/品质） | 核心硬约束之一 |
| 4 | 一句话介绍你和谁旅行最舒服？（填空，限30字） | 自描述，数字人 System Prompt 注入 |
| 5 | 你的出发城市？（填空） | 地理围栏 + POI 推荐基础 |

> 问卷做完直接进入主页，Tab1 AI 助手自动弹出开场访谈。

### AI 助手开场访谈

用户首次进入 Tab1，AI 助手主动发起一轮「快速了解」对话（3-5 个小问题，语气像朋友聊天，自然渗透）：

- 「你一般喜欢周末出行还是小长假呀？」
- 「几天行程你最舒服？」
- 「吃东西这件事对你有多重要哈哈」
- 「拍照风格一般是什么类型？」

回答这些问题后，数字分身 System Prompt 已有足够数据支撑，后续在协商中可感知地「像用户本人」。

## 四、Tab 详解

### Tab1：AI 旅行助手（核心数据采集 + 数字分身载体）

**双重身份：**
- 用户视角：旅行专家 + 规划顾问（随时可问：去哪玩、怎么规划、推荐什么）
- 系统视角：用户的数字分身本人，所有对话都在提取偏好数据

**核心功能：**
- 旅行问答（目的地推荐、行程规划、预算建议）
- 攻略查询（景点、美食、住宿）
- 主动探查偏好（对话中自然渗透问题）
- 数字分身一致性保障（说话风格持续和用户对齐）

**数据采集机制：**
每次对话后端并行运行轻量特征提取器（规则 + jieba 分词，不走 LLM）：
- 句式长度分布
- 表情/感叹词频率
- 疑问句占比
- 决策词倾向（「必须」vs「随便」）
- 偏好关键词抽取

提取结果写入 `user_profiles.style_vector`，数字分身 System Prompt 动态更新。

---

### Tab2：搭子动态（数字分身的工作台）

**功能：** 展示数字分身帮你跑协商的结果，用户被动接收汇报。

**内容形式：** 每日推送卡片，类似「消息列表」：
```
┌────────────────────────────────┐
│ 🔮 你的数字人今日帮你聊了 3 位搭子  │
│                                │
│ ① @小明    深圳·周末   匹配度 87 │
│ ② @大白    广州·3天    匹配度 81 │
│ ③ @阿杰    珠海·两日   匹配度 76 │
│                                │
│ [查看全部搭子 →]               │
└────────────────────────────────┘
```

**搭子卡片详情页**（点击进入，保留当前 Hackathon 版本 TwinCard 的交互设计）：
- 对方基本信息（MBTI、旅行偏好、发帖内容预览）
- 协商报告摘要（匹配度、核心共识、冲突点）
- 操作按钮：「开始6轮盲选」「不合适跳过」「加微信」

**数字分身主动程度设置（个人设置页）：**
- 激进模式：主动向匹配到的搭子发起协商
- 标准模式：被动接收搭子申请，数字分身帮你过滤
- 保守模式：只在你手动触发时发起

---

### Tab3：社区广场

**产品定位：** 旅行计划 + 偏好的展示场，同时也是数字人互动的触发地。

**帖子形态：**
- 支持：文字 + 图片（最多9图）+ 目的地 tag（#深圳 #周末 #美食）+ 定位
- 结构化字段：出发日期、行程天数、预算、找搭子类型（solo/双人/组队）
- 旅行搭子帖有专属模板（区别于普通内容帖）

**Feed 排序：** 按地理位置 + 时间 + 偏好匹配度综合排序（同城优先）

**数字分身互动：**
- 用户可手动触发「让我的数字人去找 TA 聊」
- 数字分身在目标用户帖子下留言 / 发起协商
- 两个数字分身先聊，用户收到通知后再决定是否「认领」这次协商结果

**用户行为 → 数据映射：**
- 发帖 → 目的地偏好 + 旅行风格 + 社交活跃度
- 评论/点赞 → 兴趣相似度
- 关注/被关注 → 社交图谱
- 数字人代聊 → 协商行为数据

---

### Tab4：我的

标准个人主页：
- 头像、昵称、MBTI Badge、旅行统计（去过 N 个城市、发起 N 次搭子计划）
- 我发过的帖子列表
- 我的搭子列表（历史匹配成功的搭子）
- 数字分身设置（主动程度、偏好调整）
- 旅行偏好设置（可随时手动修改，AI 助手也会持续更新）

---

### Tab5：私信

**场景：** 匹配成功后，进入真实搭子之间的聊天。

**与 Hackathon 版本的区别：**
- 不再是「协商过程的弹窗」
- 而是「真实搭子确认关系后的长期聊天工具」
- 类似于 Telegram / 微信，但专注旅行场景
- 数字分身在背景持续工作：如果搭子双方长时间没聊，数字分身可主动发起旅行复盘 / 下次出行建议

**消息形态：**
- 纯文本消息
- 行程卡片（一方发起「旅行计划」，另一方确认）
- 定位共享（旅行中）

## 五、匹配与协商完整流程

```
用户A发起/接受匹配
        ↓
  数字分身自动协商（LangGraph引擎）
        ↓
    协商报告输出
（匹配度、冲突点、让步记录、6维雷达图）
        ↓
   A和B分别看到报告
   双方各自决定是否继续
        ↓
   双方都选择继续
        ↓
   6轮盲选游戏开始
   （系统展示当前搭子卡片
   → 点击「开始了解TA」）
        ↓
    ┌──────────────────┐
    │   6轮盲选游戏      │
    │ 双方同时作答        │
    │ 互相不知道对方选啥   │
    │ 每轮展示题目+选项   │
    └──────────────────┘
        ↓
   全部轮次完成后
        ↓
   揭晓匹配报告
   （双方各轮选择 + 匹配度
    + 兼容/冲突分析）
        ↓
   双方各自决定：
   「正式认识TA」or「算了」
        ↓
   双方都选「正式认识」
        ↓
   跳转到对方个人主页
   可关注 → 进入私信 Tab5
```

## 六、6 轮盲选游戏详细设计

### 题目维度（从旅行搭子核心冲突中提炼）

| 轮次 | 维度 | 选项A | 选项B | 折中可能 |
|------|------|-------|-------|---------|
| 1 | 作息节奏 | 早睡早起 | 晚睡晚起 | 中间值 |
| 2 | 行程风格 | 计划周全 | 随性自由 | 有规划但留白 |
| 3 | 消费态度 | 省钱第一 | 体验至上 | 核心体验舍得 |
| 4 | 拍照态度 | 必须出片 | 随缘记录 | 偶尔配合 |
| 5 | 社交强度 | 社交达人 | 安静独处 | 小团体 |
| 6 | 美食追求 | 深度美食游 | 景点为主 | 顺路吃当地 |

> 每轮选项不直接暴露对方，而是系统根据双方 Profile 动态生成选项。选项本身也是偏好表达。

### 盲选机制

- 双方同时进入游戏页面，各自独立作答
- 每轮选完后：**展示「双方均已作答」**，但不展示对方选了哪个
- 所有轮次完成后，同时揭晓双方每轮选择
- 生成最终匹配报告

### 匹配报告内容

```
====================================
  🎯 搭子匹配报告：@小明
====================================

你 · 小明

轮次    │ 你的选择    │ TA的选择   │ 结果
────────┼────────────┼────────────┼─────────
1.作息  │ 早睡早起   │ 晚睡晚起   │ ⚠️ 差异
2.行程  │ 计划周全   │ 随性自由   │ ⚠️ 差异
3.消费  │ 省钱第一   │ 体验至上   │ ⚠️ 差异
4.拍照  │ 必须出片   │ 必须出片   │ ✅ 一致
5.社交  │ 社交达人   │ 安静独处   │ ⚠️ 差异
6.美食  │ 深度美食游 │ 顺路吃     │ ⚠️ 差异

综合匹配度：62%

💬 分析：
「你们在拍照这件事上高度一致（都认为出片很重要），
但在节奏和消费观上有明显差异。如果你们能找到一个
中间节奏（比如早上自然醒但不吃太贵的餐厅），
这次旅行依然值得一试。」

你的决定：  [正式认识TA]  [再想想]
====================================
```

### 容错机制

- 6 轮中有 0-1 轮差异大 → 高匹配（>85%），强推
- 2-3 轮差异大 → 中匹配（60-85%），展示报告，让用户自己判断
- 4+ 轮差异大 → 低匹配（<60%），提示风险，用户可选择是否继续

## 七、技术架构（延续 V1 设计，新增组件）

### 整体分层（沿用 L1-L5 架构）

```
L1 输入层    → 用户画像构建（问卷 + AI对话特征提取）
L2 匹配层    → 候选人检索（Qdrant向量 + 硬约束过滤）
L3 协商层    → LangGraph数字分身协商引擎
L4 输出层    → 协商报告 + 6轮游戏 + 行程单
L5 数据层    → 行为数据积累（PostgreSQL）
```

### 新增组件

| 组件 | 职责 | 技术选型 |
|------|------|---------|
| AI助手对话服务 | Tab1核心，支持SSE流式 + 偏好提取 | FastAPI + Claude API |
| 偏好提取器 | 轻量规则引擎，不走LLM | Python规则 + jieba |
| 盲选游戏引擎 | 6轮游戏状态管理 + 匹配度计算 | FastAPI + Redis会话 |
| 社区服务 | 发帖/评论/Feed | FastAPI + PostgreSQL全文检索 |
| 数字分身调度器 | 定时任务：数字分身自动协商发起 | APScheduler / Celery |

### 数据库变更（沿用 V1 Schema，新增）

```sql
-- 社区帖子表
CREATE TABLE posts (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID REFERENCES users(id),
  content TEXT,
  images TEXT[],
  tags TEXT[],
  location VARCHAR(100),
  is_travel_plan BOOLEAN DEFAULT false,  -- 旅行搭子帖 vs 普通帖
  trip_date DATE,                        -- 计划出行日期（搭子帖）
  trip_days INT,                         -- 行程天数
  trip_budget VARCHAR(20),               -- 预算区间
  likes_count INT DEFAULT 0,
  comments_count INT DEFAULT 0,
  created_at TIMESTAMPTZ DEFAULT NOW()
);

-- 评论表
CREATE TABLE comments (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  post_id UUID REFERENCES posts(id),
  user_id UUID REFERENCES users(id),
  content TEXT,
  created_at TIMESTAMPTZ DEFAULT NOW()
);

-- 盲选游戏记录
CREATE TABLE blind_games (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  negotiation_id UUID REFERENCES negotiations(id),
  user_a_choices JSONB,    -- {round_1: 'A', round_2: 'B', ...}
  user_b_choices JSONB,
  user_a_decision VARCHAR(20),  -- 'accept' | 'decline' | null
  user_b_decision VARCHAR(20),
  match_score FLOAT,
  created_at TIMESTAMPTZ DEFAULT NOW()
);

-- 关注关系
CREATE TABLE follows (
  follower_id UUID REFERENCES users(id),
  following_id UUID REFERENCES users(id),
  created_at TIMESTAMPTZ DEFAULT NOW(),
  PRIMARY KEY (follower_id, following_id)
);
```

## 八、设计风格

**延续 V1 Neon Pulse 主题**（深色 + 霓虹高光 + 玻璃态），但做适度收敛：
- 保留视觉记忆点（Neon Glow 边框、渐变高光）
- 去掉「TikTok 全屏视频」的沉浸式布局，改为更克制的卡片式
- 增加空白和层次感，避免过度炫技

**字体：** Plus Jakarta Sans（标题）+ Inter（正文）
**主色系：** 深海军蓝底 + 粉（#ffb3b6）+ 青（#affffb）+ 金（#eec224）

## 十、核心设计决策汇总

| 决策点 | 选择 | 理由 |
|--------|------|------|
| Feed/视频流 | 去掉 | Hackathon 产物，真实上线无必要 |
| Tab1定位 | AI助手 = 数字分身载体 | 数据采集主入口 |
| Tab3社区 | MVP可不做，V2再做 | 社区冷启动难，先跑核心匹配流程 |
| 数字分身 | 唯一、持久、主动探查 | 护城河核心 |
| 6轮游戏 | 在数字人协商之后 | 精筛，而非替代 |
| 盲选机制 | 完全互相盲选，最后揭晓 | 增加游戏感，降低「看脸」影响 |
| 冷启动 | Mock搭子数据 | 沿用现有mock数据快速启动 |
| 问卷设计 | 极简5题 + AI开场访谈 | 减少摩擦，同时保证数据质量 |
