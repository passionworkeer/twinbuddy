# PRD：孪生搭子 TwinBuddy v2

> 版本：v2.0
> 日期：2026年4月17日
> 适用：抖音赛季黑客松（40小时，4人团队）
> 赛道：赛道三 · AI体验：刷到懂你的瞬间
> 核心理念：**刷到懂你的瞬间——你的数字孪生体已在后台替你谈好了，点个头就能出发。**

**一句话定位：** 不是主动找搭子的工具，而是刷抖音时自然出现的一张懂你卡片——两个数字孪生体已经替你协商完毕，结果直接端到你面前。

**Slogan：** 你的另一个你，已经替你搞定了。

---

## 1. 产品背景与愿景

### 1.1 赛道核心要求 vs 当前产品错位

赛道三原文：

> "创造一种新的信息流内容单元……在被刷到的那一刻，把你想要的信息、情绪与选择，都刚好准备好。"

v1 PRD 设计的是"主动操作型"产品：用户上传数据 → 生成数字人 → 主动去匹配。这种形态是工具，不是"刷到的瞬间"，和赛道存在严重错位。

### 1.2 v2 改造核心

**从"主动找搭子"变为"被动懂你卡片"。**

用户正常刷抖音，突然一张 AI 卡片弹出来——你的数字孪生体已经帮你和对方聊过了，行程排好了，搭子也筛出来了，你只需要点"同意"。

关键理念转变：
- v1：给你一个工具，你自己造数字人找搭子
- v2：你刷视频时，AI 已经偷偷帮你找好搭子、协商完、排出行程了，你只需要点头

### 1.3 长期愿景

每个人都拥有一个完整人格的数字孪生体，它：

- **持续进化**：通过多源数据持续训练，精准模拟真实性格、决策风格、说话习惯、兴趣深度
- **全场景覆盖**：旅行搭子、美食搭子、赛事搭子、职场导师……所有"需要找个伴"的场景一网打尽
- **被动触发**：数字孪生体在后台持续运行，当检测到用户需求信号（刷到相关视频/内容）时，自动弹出协商结果卡片
- **主动汇报**：用户也可主动进入"我的搭子"页面，数字孪生体定期汇报"我帮你聊了 X 个搭子，以下是结果"
- **用户主权**：用户可随时"喂养"数字人（上传新数据 → 实时进化），也可随时删除

### 1.4 技术核心：MING 人格蒸馏引擎

孪生搭子的数字孪生体直接基于已有的 MING Skill（E:\desktop\hecker\MING\）构建：

- 9步灵魂锻造流程（identity → values → reasoning → boundaries → voice → relationships → growth）
- 7维度可验证的人格 distillation
- 证据驱动的持续进化
- 用户自述后亲自判断"像不像"的验证机制

**MING 是产品差异化壁垒。** 没有 MING，数字人只是标签合集；有了 MING，数字人才是"另一个你"。

### 3.3 数据输入（主动路径）

**两条数据路径并存：**

**路径A — 被动触发（主要 Demo 路径）：**
- 使用 Mock 数据：预设旅行视频内容 + 预设人格池
- 用户无需上传任何数据，直接刷 feed 体验完整流程
- 触发时机由 Mock 视频 ID 决定（第3-5条为懂你卡片触发点）

**路径B — 主动输入（用户自建数字孪生体）：**
| 数据源 | 格式 | 说明 |
|--------|------|------|
| MBTI | 文字输入 | 最快速的人格框架输入 |
| 个人简介/Bio | 文字输入 | 用户自述，主观真实 |
| AI聊天记录 | TXT/JSON | 行为真实，说话方式直接提取 |
| 照片 | 图片上传 | Avatar 生成（可选）|

**输入界面：**
- 引导页：展示支持的数据源类型
- 拖拽上传区：支持多文件同时上传
- 实时反馈：上传后显示"已识别 X 条数据，开始蒸馏……"
- 隐私提示："上传数据仅用于本次数字人生成，不会永久存储"

### 3.5 双数字人协商引擎

**使用 LangGraph 状态机控制协商流程：**

```
IDLE → PERSONA_INIT → CHAT_ROUND → CONFLICT_DETECTED →
NEGOTIATION → CONSENSUS_FOUND → REPORT_GENERATED → CARD_TRIGGERED
```

**协商内容（旅行搭子场景示例）：**
```
数字人A：我更倾向于说走就走，不想提前订太死的行程
数字人B：我需要提前知道每天住哪，临时找酒店我会焦虑
→ 协商：A同意提前订住宿，B同意不锁死每天具体行程
→ 共识：行程大框架提前定，细节弹性
→ 卡片展示："你们的旅行节奏兼容度78%，已在行程中留出弹性时间"
```

**对方数字人的生成策略：**
```
Demo期（黑客松）：
  预设4个Mock人格（见3.6节）
  你的孪生体 × Mock人格 → 协商 → 卡片展示结果

未来（真实用户接入后）：
  你的孪生体 × 对方真实数字孪生体 → 真实协商
  架构不变，只需接入真实用户数据
```

### 5.4 MING 蒸馏 Prompt

```python
ming_fusion_prompt = """
你是MING人格蒸馏引擎。

【输入数据】
MBTI: {mbti}
个人简介: {bio}
聊天记录片段: {chat_history}

【任务】
基于上述数据，生成完整的persona JSON和数字人自述文本。

【人格画像生成规则】
1. 从聊天记录中提取真实的说话风格（词汇、短句模式、口头禅）
2. MBTI作为人格骨架，填充具体行为证据
3. Bio作为主观自述基准，对齐或补充聊天记录
4. 所有推断必须有数据依据，不凭空捏造

【自述文本规则】
- 第一人称，2-4段
- 包含具体例子（从聊天记录中来）
- 体现人格的核心矛盾（e.g., "我不爱计划，但我讨厌失望"）
- 语气必须和聊天记录中的用户语气一致

输出JSON + 自述文本，不要其他解释。
"""
```

### 5.5 隐私设计

| 原则 | 实现方式 |
|------|---------|
| 数据不过留 | 所有上传数据仅用于本次会话，会话结束即清理 |
| 用户主权 | 用户可随时触发"删除我的数字人" |
| 透明告知 | 上传页醒目展示隐私说明 |
| 临时存储 | Session级别存储，不落持久化数据库 |

## 9. 风险与应对

| 风险 | 等级 | 应对 |
|------|------|------|
| 评委质疑"这不是真正的抖音信息流" | 中 | 明确说这是"Demo展示形态"，核心是信息流卡片体验，真实落地会在抖音生态内 |
| LLM调用超时/失败 | 高 | 预生成协商结果降级展示；超时提示"数字人思考中" |
| Feed滑动体验不够流畅 | 中 | 提前调试；可改为自动播放+卡片自动弹出，无需手动滑动 |
| 评委想体验上传数据生成数字人 | 低 | 保留主动上传入口作为次要功能，但主流程不需要 |
| 界面在大屏上显示异常 | 中 | 提前调试大屏分辨率；CSS响应式降级 |

## 附录：v1 → v2 变更说明

| 变更项 | v1 设计 | v2 设计 | 变更原因 |
|--------|---------|---------|---------|
| 产品形态 | 独立工具，用户主动操作 | 模拟抖音Feed，被动触发 | 匹配赛道三"信息流内容单元" |
| 核心入口 | 数字人自述验证页 | 懂你卡片（刷feed时弹出） | 降低用户操作成本，提高惊艳感 |

| 场景聚焦 | 通用搭子（旅行/美食/职场） | 旅行搭子为主，Demo展示3-5条Feed | 40小时聚焦一个场景打出冲击力 |
