你上一次找人一起旅游，是什么时候？发了几十条消息？协调了多少次？

我有过这种经历。出发前两周开始找人，好不容易凑齐人，行程谈不拢、预算对不上、有人想早起有人想赖床……

**找人这件事，比旅游本身还累。**

我们今天带来的产品叫 **TwinBuddy 孪生搭子**。

我们做了一件事——

**你的数字孪生体，在后台帮你把搭子找好了、谈好了、行程全定好了。**

你只需要点一个「同意」。

## 三、Step 1 · MBTI 选择（预计 30 秒）

---

**【演示：点击 ENFP】**

第一步，选你的 MBTI 人格类型。

十六种全覆盖——这是我们整个系统的认知基础。

你的 MBTI 决定了数字孪生体"怎么想问题"。

选完之后，系统自动跳到下一步。

---

## 四、Step 2 · 兴趣标签（预计 30 秒）

---

**【演示：多选 4-5 个标签，如"美食""打卡""拍照""公路旅行"】**

第二步，选你喜欢的旅行方式。

可以多选——这些标签决定了你的数字孪生体"怎么行为"。

比如你选了"美食"和"慢节奏"——数字人格就会知道，你是一个旅行中不愿赶路、以吃为主的人。

## 五、Step 3 · 语音 / 文字描述（预计 45 秒）

---

**【演示：点击麦克风，说一句话，如"我想找一个说走就走、不喜欢计划得太死的人"】**

第三步，最关键的一步——用一句话描述你理想的搭子。

这里有两个输入方式：

**文字输入**：直接打字。

**语音输入**：点击麦克风，直接说话。

我们接入了 **iFlytek 实时语音识别**——你说话的同时，文字实时转写上来。

> **【演示：对着麦克风说话，看到文字实时出现】**

可以看到，文字是逐字出现的，不是一句话说完才出来。

这就是你在"训练"你的数字孪生体——告诉它你是什么样的人，它才能替你去谈判。

> **【可选：演示修改已识别文字】**
> 你可以在文字上直接修改，调整到满意为止。

### 10.1 六维雷达图

**【指着雷达图】**

先看这张雷达图——六个维度：

| 维度 | 含义 |
|------|------|
| 节奏 | 暴走派 vs. 慢悠派 |
| 美食 | 吃货 vs. 随便吃点 |
| 拍照 | 出片狂魔 vs. 不拍也行 |
| 预算 | 品质优先 vs. 能省则省 |
| 冒险 | 挑战极限 vs. 安全第一 |
| 随性 | 说走就走 vs. 计划周全 |

中间绿色的数字是**旅行契合度**——这张图里是 88%。

绿色多边形越接近外圈，两个人越合拍。

### 10.2 契合度 + 标签

这三个标签，是系统从协商对话中**自动提取出来的共识**，不是预设模板。

这是真实的两轮对话摘要——你的数字孪生体怎么说、对方怎么回应。

## 十二、技术讲解（预计 60 秒）

---

> 本节面向技术评委，非技术评委可跳过直接到愿景部分。

---

### 12.1 MING 人格蒸馏

用户的四个输入——MBTI + 兴趣标签 + 文字描述 + 目的地——传入 **MiniMax LLM**，用 MING 四维度 prompt 约束系统，生成完整人格文档。

不是模板填充，是 LLM 生成。

人格文档保存在客户端 localStorage，不上传服务器。

---

### 12.2 预计算机制

用户选择城市的那一刻，**后台就开始预计算**：

```
选城市 → 启动预计算
           ├── 获取 top1 最契合搭子
           ├── 生成对方人格文档
           └── 运行双 Agent 协商（实时调用 MiniMax LLM）
```

等用户在 Feed 里刷到第 2-3 条视频时，预计算结果已经就绪，**零等待**弹出懂你卡片。

---

### 12.3 双 Agent 协商状态机

两个数字人，各自是一个 LLM Agent。

用 **LangGraph 状态机**控制协商流程：

```
提案 → 评估 → 让步/拒绝 → 共识 / 继续下一轮
```

最多三轮，每一轮都是真实的 LLM 对话。

协商结果包含：目的地、天数、预算、行程标签、雷达数据、分歧点。

---

### 12.4 iFlytek 实时语音识别

前端通过 **WebSocket** 连接 iFlytek 实时 STT 服务。

音频流：`麦克风 → AudioWorklet（PCM 处理）→ WebSocket → iFlytek → 文字`

识别结果实时回写 textarea，用户可边说边改。

---

### 12.5 技术栈一览

| 层级 | 技术 |
|------|------|
| 前端 | React + Vite + Tailwind CSS + TypeScript |
| 后端 | FastAPI + Python |
| Agent | LangGraph 状态机 |
| LLM | MiniMax（M2）|
| 语音 | iFlytek WebSocket STT |
| 部署 | Railway（后端）+ Vercel（前端）|

# 附录 B：评委常见问题 Q&A

| 问题 | 回答 |
|------|------|
| 人格是怎么生成的？ | MiniMax LLM + MING 四维度 prompt，不是模板填充 |
| 两个 Agent 怎么谈判？ | LangGraph 状态机，最多三轮，每轮都是 LLM 真实对话 |
| 评分是怎么算的？ | 六维度加权平均，总分 0-100，80+ 才触发懂你卡片 |
| 和普通匹配有什么区别？ | 普通匹配是筛选，我们的是谈判——两个 AI 替你达成共识 |
| 数据安全吗？ | 人格文档存 localStorage，不上传服务器 |
| 为什么这么快？ | 用户选城市时，后台已经预计算好了 |
| 语音识别用了什么？ | iFlytek 实时 WebSocket STT，AudioWorklet 处理 PCM 流 |
| 跳转到抖音是什么？ | 随机选一条抖音视频链接，引导加好友 |

# 附录 C：演示异常处理预案

| 情况 | 处理方式 |
|------|---------|
| 评委手机扫码打不开 | 切换到平板演示，或直接用自己手机投影 |
| Feed 里没有触发懂你卡片 | 引导多刷几条，第 2-3 条必触发（代码设定） |
| 懂你卡片加载很慢 | 解释：后端在跑 MiniMax LLM 协商，通常 5-8 秒 |
| 语音识别失败（浏览器不支持） | 切换到文字输入模式，产品支持两种方式 |
| 网络不稳定 | 有 Mock 降级，协商用预设对话代替 LLM 调用 |
| 高分评委问商业化 | 抖音数字孪生体生态 + 搭子付费场景 + B 端团建 |

# 附录 D：产品 UI 细节备忘

> 讲解时可以用这些具体细节增加真实感

- **Onboarding 背景**：一张城市风景背景图，第四步时变暗并模糊
- **MBTI 按钮样式**：选中时有粉色霓虹光晕（`shadow-[0_0_10px_rgba(255,181,159,0.4)]`）
- **语音波形动画**：8 根柱子，随 sin 函数动态高度变化
- **城市卡片**：横向滚动，IntersectionObserver 自动居中选中的卡片
- **Feed 顶部 Nav**：模拟抖音，有"团购/经验/北京/关注/推荐"Tab
- **loading 界面**：绿色圆角气泡代表搭子，白色消息条代表你
- **雷达图**：SVG 手绘六边形，绿色多边形填充，数据点白色圆点
- **击掌动画**：highfive.webm 全屏播放，播放完跳转抖音链接
- **greeting 动画**：greeting.webm 在卡片弹出时从底部弹出，4.4 秒后消失
